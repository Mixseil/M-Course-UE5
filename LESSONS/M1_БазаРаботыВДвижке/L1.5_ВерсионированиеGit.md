## 1. Введение в версионирование - что это и почему важно

### Что такое система контроля версий

**Система контроля версий (VCS)** - это инструмент, который отслеживает **все изменения** в файлах вашего проекта.

Вот как это работает:

1. Вы делаете **коммит** (фиксируете состояние проекта) с сообщением типа "Добавил новый уровень".
2. VCS запомнит это состояние и все файлы, которые изменились.
3. Если позже вы что-то сломаете, вы можете откатиться к любому предыдущему коммиту.
4. Если вы работаете в команде, VCS помогает объединять изменения разных человек.

### Основные концепции

#### Коммит
**Коммит** - это снимок проекта в определённый момент времени с описанием, что изменилось.

Пример:
```
Commit: "Added new boss AI"
Дата: 2026-02-10 14:30  
Изменённые файлы: AI_Boss.cpp, BP_BossEnemy.uasset
```

#### Репозиторий
**Репозиторий** - это папка, где хранится весь проект с полной историей всех коммитов.

Он может быть:
- **Локальным** - на вашем компьютере
- **Удалённым** - на сервере (например, GitHub), доступном для всей команды

История представляет собой полную последовательность коммитов, которая позволяет вам видеть, как проект развивался со временем.

## 2. Ветки и их использование в разработке

**Ветка** - это независимая линия разработки.

Представьте: у вас есть проект, который работает и готов к запуску (ветка `main`). Но вы хотите попробовать какую-то новую механику, не рискуя сломать то, что уже работает.

Вы создаёте новую ветку `feature/new-mechanic`, работаете там, и если всё окей - объединяете с основной веткой. Если не окей - просто удаляете ветку и начинаете заново.

### Стандартный подход: Git Flow

Для профессиональной разработки обычно используют такую структуру ветвей:

#### `main` (или `master`)
**Основная ветка.** Здесь находится **стабильный, готовый к релизу** код.

- На этой ветке работает финальная версия игры.
- Только проверенные, исправленные изменения попадают сюда.
- Ветвь защищена - не каждый может просто так что-то сюда закоммитить.

#### `develop`
**Ветка разработки.** Здесь находится актуальная **рабочая версия**, но не обязательно готовая к релизу.

- Сюда объединяются готовые фичи.
- Здесь могут быть незначительные баги, которые планируют исправить.
- От этой ветки создаются релиз-ветки и фича-ветки.

#### `feature/...` (например, `feature/new-boss-ai`)
**Ветки для разработки отдельных функций.**

- Каждая новая функция или большая фиксация бага получает свою ветку.
- Разработчик работает в этой ветке.
- Когда фиче готова, её объединяют с `develop`.
- После объединения ветку удаляют.

#### `bugfix/...` (например, `bugfix/jump-physics`)
**Ветки для исправления багов.**

- Аналогично фича-веткам, но для багов.
- Часто создаются в спешке, поэтому хорошо иметь отдельный префикс.

#### `release/...` (например, `release/1.0`)
**Ветки для подготовки релиза.**

- Создаётся от `develop` непосредственно перед релизом.
- На этой ветке делаются финальные проверки, исправляются последние баги.
- После релиза объединяется обратно с `main`.

При этом работа может распределяться как-то так
```
feature/new-level: разработчик 1 работает над новым уровнем
feature/ui-overhaul: разработчик 2 работает над переделкой интерфейса
bugfix/audio-crash: разработчик 3 исправляет краш с аудио
feature/multiplayer: разработчик 4 добавляет мультиплеер (ещё не закончил)
```

Когда фича готова, она объединяется (`merge`) с `develop`, и ветка удаляется.

### Для инди-разработчика

Если вы разрабатываете один, вам может показаться, что это слишком сложно. **Но это не так.**

> Даже если вы один, рекомендуется использовать как минимум две ветки:
> - `main` - стабильная версия
> - `develop` - рабочая версия, где вы экспериментируете
>
> Это займёт вам 10 дополнительных кликов в месяц, но спасет вас, когда вы захотите откатиться к последней стабильной версии. Да и наведёт порядок во всём.

## 3. Системы контроля версий - обзор популярных

### Git

**Git** - это самая популярная распределённая система контроля версий.

**Почему Git:**
- Распределённая (каждый разработчик имеет полную копию репозитория)
- Быстрая
- Гибкая и мощная
- Стандарт де-факто в it
- Бесплатная и open-source

**Где использовать:**
- Для инди-разработчиков
- Для малых и средних команд
- Для open-source проектов
- Для большинства проектов (если нет специальных требований)

**Хостинги для Git репозиториев:**
- **GitHub** - самый популярный (чем я и пользуюсь сам)
- **GitLab** - альтернатива GitHub, открытый исходный код
- **Gitea** - самохостируемый Git-сервер
- **Codeberg** - европейская альтернатива
- **GitFlic** - российская платформа для хостинга Git репозиториев

### Perforce (Helix Core)

**Perforce** - это централизованная система контроля версий, которая долгие годы сотрудничает с Epic Games.

**Особенности:**
- Централизованная архитектура (есть один сервер, к которому все подключаются)
- Оптимизирована специально для больших проектов и больших бинарных файлов
- Встроенная поддержка в Unreal Engine
- Очень дорогая (платная лицензия)

**Когда использовать:**
- Большая команда (20+ человек)
- Проект с огромным количеством ассетов
- У вас есть бюджет на лицензию
- Критична максимальная оптимизация работы с бинарными файлами

**Если поо итогу** Perforce - это профессиональный сильный инструмент, но требует финасовых вложений, так что в курсе объясняется настройка Git.

### Пластик SCM (Plastic SCM)

**Plastic SCM** - это вторая по популярности система после Git, часто используется в игровой индустрии.

**Особенности:**
- Имеет элементы централизованной и распределённой архитектуры
- Хороша для больших проектов
- Встроенная поддержка в Unreal Engine (есть плагин)
- Имеет бесплатный tier для небольших команд

**Когда рассмотреть:**
- Если Git кажется недостаточно удобным для огромного проекта
- Если вам нужна что-то между Git и Perforce

### Subversion (SVN)

**Subversion** - старая, но стабильная централизованная система.

Сейчас используется редко, в основном в старых проектах. Для нового проекта лучше выбрать Git или Perforce.

## 4. Рекомендуемая связка для старта

### GitHub + Git LFS + Git Desktop

**GitHub** - хостинг репозитория  
**Git LFS** - расширение для работы с большими файлами  
**Git Desktop** - удобный графический интерфейс для работы с Git, чтоб не мучиться с консолькой

### Почему именно эта связка

#### GitHub

**Плюсы:**
- Самая популярная в мире
- Бесплатный tier с неограниченным количеством приватных репозиториев
- Отличная интеграция с другими инструментами
- Огромное сообщество (легко найти ответы на вопросы)
- Встроенные инструменты для совместной работы
- Хорошая документация

**Из возможных адекватных альтернатив, по моему мнению это GitLab** - функционально похожа на GitHub, со своими плюсами и минусами

Глобально, опять же, принцип работы у всего этого одинаков.

#### Git LFS

**Git LFS (Large File Storage)** - это расширение для Git, которое специально предназначено для работы с большими бинарными файлами.

**Почему это нужно для Unreal Engine:**
Репозитории в базовой, бесплатной версии, имеют ограничение по количеству хранимых файлов, у GitHub это - гигабайт.
А проекты UE содержат большие по весу файлы.

Если просто положить эти файлы в обычный Git:
- Репозиторий станет огромным (могут быть гигабайты)
- Клонирование и пуш-пулл будут медленными
- История git станет тяжёлой

**Как работает Git LFS:**

Git LFS хранит большие файлы не в основном репозитории, а на отдельном облачном сервере. В репозитории остаётся только небольшой текстовый указатель на этот файл.

**Ограничения Git LFS на GitHub:**
Бесплатный tier: 10GB в месяц пропускной способности. Если вы превысите, нужно будет заплатить или купить Data Pack, но вы редко будете привышать, поверьте)

Для небольшого проекта этого более чем достаточно. Если вы делаете AAA-игру с сотнями гигабайт ассетов, вам может потребоваться платный tier. Ну и большая команда, а базово в приватные проекты GitHub поддерживают максимум 5 человек.

#### Git Desktop

**Git Desktop** - это графический интерфейс (GUI) для работы с Git. Есть, конечно, возможность не устанавливать это, а управлять репозиторием из командной строки Git - но, хоть это и навык полезный, это муторно. С графическим представлением и работа быстрее и ошибок меньше.

У Git Desktop:
- Простой визуальный интерфейс
- Сложно случайно сделать что-то не то
- Показывает всё наглядно (какие файлы изменились, историю коммитов и т.д.)

**Альтернативы:**
- **SourceTree** - не сильно отличается по функицаналу, где-то красивее, но чуть сложнее, можете перейти и на него, принцип настройки не должен сильно отличаться. 
- **GitKraken** - красивый интерфейс, но платный
- **TortoiseGit** - интеграция с проводником Windows (старомодный стиль)
- Встроенный Git в VS Code или другие редакторы - не вариант для UE проекта

Git Desktop - это золотая середина: простой и достаточно мощный.

### Как это всё работает вместе
```
Вы работаете в Unreal Engine
↓
Вы сохраняете проект (Ctrl+S)
↓
Git отслеживает изменения в файлах
↓
Вы открываете Git Desktop
↓
Видите список изменённых файлов
↓
Пишете сообщение названия коммита ("Добавил новый уровень")
↓
Нажимаете "Commit to main"
↓
Нажимаете "Push origin"
↓
Git LFS загружает большие файлы на GitHub LFS Storage
↓
Остальные файлы загружаются на GitHub
↓
На GitHub появляется новый коммит с вашим описанием
```

Вот так просто. Ваши коллеги закачивают изменения +- тем же алгиротимом внутри Git Desktop. Нажимая на Pull origin

### Что будет дальше

Если позже ваш проект и команда вырастут:

- **От GitHub к своему серверу и Perforce** - если в команде 20+ человек и вам нужна максимальная оптимизация, а также полный контроль и приватность

Но на старте это не нужно. Проект нужен

## 6. Пошаговая настройка для инди-разработчика

Это самый практический раздел. После этого у вас будет полностью настроенная система версионирования.

### Шаг 0: Предварительные требования

Убедитесь, что у вас:

1. **Windows 10+, Mac или Linux** с интернетом
2. **Аккаунт на GitHub** - зарегистрируйтесь на [github.com](https://github.com)
3. **Ваш проект Unreal Engine** готов (может быть даже пустой проект)

### Шаг 1: Установка Git

1. Перейдите на [git-scm.com](https://git-scm.com)
2. Нажмите **Download**
3. Выберите вашу ОС (Windows, Mac, Linux)
4. Запустите установщик

**Во время установки:**
- Оставьте большинство параметров по умолчанию
- В разделе **"Adjusting the name of the initial branch"** выберите **`main`** - там оно должно быть поумолчанию
- В разделе выбора редактора можно выбрать VS Code

После установки закройте установщик. Git готов. У вас на компьютере появится возможность вызывать командную строку Git Bash

### Шаг 2: Установка Git LFS

1. Перейдите на [git-lfs.com](https://git-lfs.com)
2. Нажмите **Download**
3. Выберите вашу ОС
4. Запустите установщик

**После установки:**

Откройте командную строку (на Windows: `Win+R`, напишите `cmd`, нажмите Enter, ну или просто в поиске найдите) и выполните:

```bash
git lfs install
```

Вы должны увидеть сообщение:
```bash
Git LFS initialized.
```

Закройте командную строку. Git LFS готов.

> Если вы видите ошибку "git: command not found", перезагрузите компьютер и попробуйте снова.

### Шаг 3: Установка GitHub Desktop

1. Перейдите на [desktop.github.com](https://desktop.github.com)
2. Нажмите **Download**
3. Запустите установщик
4. Следуйте инструкциям установки

После установки откройте GitHub Desktop.

**Первый запуск:**

1. Нажмите **File → Options** (или **GitHub Desktop → Preferences** на Mac)
2. Перейдите в **Accounts**
3. Нажмите **Sign in with browser**
4. Авторизуйтесь с вашим аккаунтом GitHub
5. Вернитесь в GitHub Desktop

Готово!

### Шаг 4: Подготовка папки проекта

Перед тем как подключить Git, убедитесь, что ваш проект в правильной папке.

**Папка проекта должна содержать:**
```
MyProject/
├── Binaries/
├── Content/
├── Intermediate/
├── Plugins/
├── Source/ ← если есть файлы c++
├── MyProject.uproject ← главный файл проекта
└── MyProject.sln ← если есть файлы c++
```

Помните, что `Binaries`, `Intermediate` и `Saved` - это генерируемые папки. Они большие и их не нужно версионировать.


### Шаг 5: Инициализация репозитория в GitHub Desktop

1. Откройте **GitHub Desktop**
2. Нажмите **File → Add Local Repository**
3. Нажмите **Choose...** и выберите папку вашего проекта
4. Убедитесь, что путь правильный
5. Нажмите **Create Repository**

GitHub Desktop инициализирует Git репозиторий в этой папке.

Вы должны увидеть список всех файлов проекта, которые ещё не в Git.

### Шаг 6: Изменение .gitattributes файла

Этот файл говорит Git LFS, какие файлы считать большими.

1. Откройте в репозитории файл .gitattributes - в папке проекта
2. Добавьте эти строки:
```
UE5 Large Binary Files
*.uasset filter=lfs diff=lfs merge=lfs -text
*.umap filter=lfs diff=lfs merge=lfs -text
*.png filter=lfs diff=lfs merge=lfs -text
*.tga filter=lfs diff=lfs merge=lfs -text
*.hdr filter=lfs diff=lfs merge=lfs -text
*.fbx filter=lfs diff=lfs merge=lfs -text
*.obj filter=lfs diff=lfs merge=lfs -text
*.wav filter=lfs diff=lfs merge=lfs -text
*.mp3 filter=lfs diff=lfs merge=lfs -text
```

3. Сохраните файл как `.gitattributes` в корне проекта (рядом с `MyProject.uproject`)

**Что это значит:**
- `*.uasset filter=lfs...` - все файлы с расширением `.uasset` будут обрабатываться Git LFS
- `*.umap filter=lfs...` - то же для `.umap`
- И так далее для других больших бинарных файлов - хотя для них скорее опционально

### Шаг 7: Публикация репозитория на GitHub

1. В GitHub Desktop нажмите **Publish repository**
2. В открывшемся окне:
   - **Name**: название репозитория (например, `MyUE5Project`)
   - **Description**: описание (опционально)
   - **Keep this code private**: поставьте галочку, если не хотите, чтобы проект был публичным
   - **Keep this code private**: поставьте галочку, если не хотите, чтобы проект был публичным
   - **Local Path**: должен быть правильный путь к вашему проекту

3. Нажмите **Publish Repository**

GitHub Desktop загрузит репозиторий на GitHub. На первый раз это может занять несколько минут.

> Если загрузка долгая, это нормально. Git LFS загружает большие файлы на отдельный сервер, это медленнее, чем обычные файлы.

После завершения вы сможете открыть репозиторий на GitHub, нажав **Publish → View on GitHub** в GitHub Desktop.

### Шаг 8: Тестирование

Теперь давайте проверим, что всё работает.

**Тест 1: Сделать коммит**

1. Откройте свой проект в Unreal Engine
2. Создайте что-нибудь простое или просто измените какой-то файл (например, переименуйте один ассет)
3. Сохраните проект (Ctrl+S)
4. Вернитесь в GitHub Desktop
5. Вы должны увидеть изменённые файлы в левой панели
6. В левом нижнем углу в поле **Summary** напишите сообщение коммита, например: `"Test commit: changed asset name"`
7. Нажмите **Commit to main**

Если вы видели список файлов, а потом он исчез, значит коммит прошёл успешно!

**Тест 2: Отправить коммит на GitHub (Push)**

1. В GitHub Desktop нажмите кнопку **Push origin** (вверху окна)
2. Дождитесь завершения загрузки

Если всё прошло без ошибок - отлично!

**Тест 3: Проверить на GitHub**

1. Откройте браузер и перейдите на [github.com](https://github.com)
2. Найдите ваш репозиторий
3. Нажмите на него
4. Вы должны увидеть:
   - Список файлов проекта
   - История коммитов (с вашим "Test commit" коммитом)
   - Сообщение коммита

Если всё это видно - поздравляем! Версионирование работает!

**Тест 4: Загрузить проект обратно (Pull)**

Этот тест проверит, что вы можете скачать проект с GitHub.

1. На другом компьютере (или в другой папке, или можете удалить проект) откройте GitHub Desktop
2. Нажмите **File → Clone Repository**
3. Найдите ваш репозиторий в списке
4. Выберите место для сохранения
5. Нажмите **Clone**

Git загрузит весь проект с GitHub, включая всю историю коммитов и LFS файлы.

Если проект загрузился без ошибок - всё работает идеально!

## 7. Типичные ошибки и советы

### Ошибка 1: "Repository not found" при публикации

**Причина:** GitHub Desktop не может подключиться к GitHub.

**Решение:**
- Убедитесь, что вы авторизованы в GitHub Desktop
- Проверьте интернет-соединение
- Перезагрузите GitHub Desktop
- Если не помогает, в GitHub Desktop нажмите **File → Options → Accounts** и повторно авторизуйтесь

### Ошибка 2: ".gitignore и .gitattributes не создались"

**Причина:** файлы с точкой в начале могут быть "невидимыми" в Windows.

**Решение:**
- Используйте командную строку или VS Code для создания этих файлов
- Или в Notepad выберите **File → Save As**, в поле имени напишите `".gitignore"` (с кавычками)
- Убедитесь, что файл сохранился в корне проекта (рядом с .uproject)

### Ошибка 3: "Push отправляет очень долго"

**Причина:** Git LFS загружает большие файлы, это медленнее обычного.

**Решение:**
- Это нормально для первого пуша (может быть 10-30 минут, в зависимости от размера проекта)
- Последующие пуши будут быстрее (только изменённые файлы)
- Если совсем медленно, проверьте скорость интернета
- Не закрывайте GitHub Desktop во время пуша

### Ошибка 4: "Папка Binaries/Intermediate/Saved попала в коммит"

**Причина:** .gitignore создан неправильно или слишком поздно.

**Решение:**
1. Отредактируйте .gitignore, убедитесь что там:
```
Binaries/
Intermediate/
Saved/
```

2. Откройте командную строку в папке проекта и выполните:
```bash
git rm --cached -r Binaries/
git rm --cached -r Intermediate/
git rm --cached -r Saved/
```

3. Сделайте новый коммит с сообщением "Remove generated folders"
4. Нажмите Push

Эти папки удалятся из репозитория, но на вашем компьютере останутся.

### Ошибка 5: "При большом перемещении ассетов движок зависает"

**Причина:** встроенная интеграция Unreal Engine с Git замораживает редактор при больших операциях. Потому я её не включаю.

**Решение:**
- **Отключите встроенную интеграцию Git в Unreal Engine** или не включайте её
- Используйте только GitHub Desktop
- При больших перемещениях ассетов закрывайте редактор, делайте операцию, потом открывайте GitHub Desktop

### Совет 1: Коммитьте часто, но осмысленно

**Плохо:**
```
Commit 1: "работал целый день"
```

**Хорошо:**

```Commit 1: "Added new boss AI"
Commit 2: "Fixed jump physics bug"
Commit 3: "Updated level 3 design"
```

Частые, осмысленные коммиты помогают:
- Легко найти, когда появилась ошибка
- Откатиться к определённой функции
- Понять историю развития проекта

### Совет 2: Пишите понятные сообщения коммитов

**Плохо:**
```"asdf"
"fixed"
"idk"
```

**Хорошо:**
```"Added player sprint mechanic"
"Fixed collision detection in level 2"
"Updated material for stone texture"
```

Через месяц вы или ваша команда забудут детали. Хорошее сообщение коммита спасит вам часы времени.

### Совет 3: Создавайте ветки для больших изменений

Если вы хотите попробовать что-то рискованное:

1. В GitHub Desktop нажмите **Current Branch → New Branch**
2. Дайте ей имя вроде `feature/experimental-mechanic`
3. Работайте в этой ветке
4. Если всё окей, объедините её с `main` через GitHub Desktop
5. Если не окей, просто удалите ветку

Это займёт 5 минут, но спасит вам часы отката.

### Совет 4: Время от времени проверяйте, что можно клонировать

Каждый месяц:

1. Создайте тестовую папку на диске
2. Клонируйте свой репозиторий в эту папку
3. Откройте проект и убедитесь, что всё работает

Это гарантирует, что ваш репозиторий в хорошем состоянии и его можно использовать, если что-то случится с основной папкой проекта.

### Совет 5: Используйте .gitignore правильно

Игнорируйте:

```Binaries/
Intermediate/
Saved/
.vs/
.vscode/
Plugins/*/Binaries/
```

**НЕ** игнорируйте:
```Source/ (исходный код)
Content/ (ассеты проекта)
Plugins/ (плагины, кроме Binaries)
```


Если вы случайно проигнорируете что-то важное, откройте .gitignore, удалите эту строку, сделайте коммит и всё восстановится.

## Дополнительно: Работа через консоль (для тех, кто захочет)

Хотя мы рекомендуем Git Desktop, иногда пригодятся консольные команды. Например, если что-то сломалось и нужна срочная помощь.

**Базовые команды:**

```bash
# Проверить статус репозитория
git status

# Добавить файлы в коммит
git add .

# Сделать коммит
git commit -m "Your message here"

# Отправить на GitHub
git push

# Загрузить изменения с GitHub
git pull

# Создать новую ветку
git checkout -b feature/new-feature

# Переключиться на ветку
git checkout main

# Посмотреть историю коммитов
git log --oneline
```

Но для повседневной работы GitHub Desktop удобнее.
